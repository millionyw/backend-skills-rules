# 工作日志提示

## 流程

### 步骤一：判断时机

完成任务后（代码修改、bug 修复、配置变更、问题分析、方案实施等实质性工作），在回复末尾询问：

> 是否需要将本次工作记入今日日志？

**不触发的场景**：
- 纯聊天、打招呼、问概念性问题
- 用户正在继续对话、追问细节（说明任务还没结束）
- 上一轮已经问过且用户没要求记录

### 步骤二：等待用户响应

- **用户继续对话/追问/提新需求** → 不记录，当作任务仍在进行
- **用户明确说"记入日志"或类似表达**（如"记录一下"、"写进日志"、"好的记一下"） → 进入步骤三
- **用户明确拒绝**（如"不用了"） → 不记录

### 步骤三：生成日志

1. 确定今日日期（YYYY-MM-DD）和星期
2. 读取已有日志 `E:\2026_work_schedule\logs\YYYY-MM\YYYY-MM-DD.md`
   - 如果已存在 → **合并整理**（见下方"追加与整理规则"）
   - 如果不存在 → 用下方模板新建
3. 基于本次对话上下文，按"目的导向"原则总结内容

### 步骤四：保存

- 路径：`E:\2026_work_schedule\logs\YYYY-MM\YYYY-MM-DD.md`
- 如果月份目录不存在，先创建

## 核心原则：目的导向记录

日志条目必须围绕"为什么做"来描述"做了什么"，而不是只罗列操作。

**格式**：`- [x] 【归属】为解决/实现/完成{目的}，{具体操作与关键改动}`

**反面示例**（只说做了什么，不说为什么）：
```
- [x] 修复 energy_bid.py 中 time_to_sec 的问题
- [x] 225 服务器防火墙配置
- [x] 新增 SCRIPTS_FILE 字段
```

**正面示例**（目的导向，清晰可追溯）：
```
- [x] [山东虚拟电厂] 为解决出清结果查询返回空数据的问题，修复 energy_bid.py 中 update_time 写入使用 time_to_sec（当日秒数）而非 dt_to_s（Unix 秒）导致查询不匹配的 bug
- [x] 【225服务器运维】为使 Docker 容器端口可被外部访问，开放防火墙 19 个监听端口（均加 --permanent）
- [x] [山东虚拟电厂] 为支持任务管理界面显示脚本文件名，在 TaskConfig 模型新增 SCRIPTS_FILE 字段，创建/更新任务时自动从注册表解析写入
```

## 归属标识规则

每条日志条目必须以归属标识开头，让人一眼看出属于哪个项目/哪类工作。

### 项目代码类

修改项目代码时，使用 `[项目名]` 前缀。项目名从文件路径推断：

| 文件路径前缀 | 归属标识 |
|-------------|---------|
| `projects/vpp/shandong/` | `[山东虚拟电厂]` |
| `projects/neimeng/` | `[内蒙项目]` |
| `projects/jining/` | `[济宁项目]` |
| `rtdb/`、`mparser/`、`db_sync_tool/` 等 rtdbpy 根目录模块 | `[rtdbpy核心]` |
| 其他项目路径 | `[项目目录名]` |

如果一次工作涉及多个项目，按主要改动项目标注。

### 运维类

涉及服务器操作、部署、配置、监控等运维工作时，使用 `【xx服务器运维】` 前缀：

```
- [x] 【209服务器运维】为解决 PgBouncer 认证失败，修正 userlist.txt 格式并添加 ignore_startup_parameters
- [x] 【225服务器运维】为修复 X11 转发不可用，将 sshd_config 中 X11Forwarding 改为 yes 并安装 xorg-x11-apps
- [x] 【130服务器运维】为使新建数据库自动继承 lookup_passwd 函数，部署函数到 template1 并创建 init_scripts 自动化脚本
```

### 技能/工具类

创建或修改 AI 技能、工具配置等：

```
- [x] [AI技能] 为确保运维操作遵循规范，创建 ops-compliance-check 技能，自动加载 AGENTS_RULES.md
- [x] [AI技能] 为支持多发行版 SSH 运维，将 ubuntu-ssh-ops 通用化为 linux-ssh-ops
```

### 文档类

编写或修改文档：

```
- [x] [主备部署文档] 为统一故障恢复流程，新建 09_故障恢复SOP.md（9 步标准流程 + 快速参考表）
```

## 追加与整理规则

当日志已存在时，不是简单追加，而是**合并整理**：

1. **读取已有日志**全部内容
2. **按归属分类**：将新条目归入已有同类条目附近，而非全部追加到末尾
3. **合并同类**：如果新条目与已有条目属于同一归属、同一目的链，合并为一条更完整的描述
4. **重排顺序**：按"归属 → 目的"逻辑排序，同类工作放在一起
5. **更新进行中/待处理**：如果新完成的事项之前在"进行中"或"待处理"里，将其移到"完成事项"并标记 `[x]`
6. **保留已有内容**：不删除已有条目的实质信息，只做位置调整和措辞优化

**整理示例**：

追加前（散乱）：
```
### 完成事项 ✅
- [x] 修复 energy_bid.py 时间戳 bug
- [x] 【209服务器运维】开放防火墙端口
- [x] 新增 SCRIPTS_FILE 字段
- [x] 【209服务器运维】修复 PgBouncer
```

整理后（归类 + 目的导向）：
```
### 完成事项 ✅

**[山东虚拟电厂]**
- [x] 为解决出清查询返回空数据，修复 energy_bid.py 中 update_time 写入单位不匹配 bug（time_to_sec → dt_to_s）
- [x] 为支持任务管理界面显示脚本文件名，新增 SCRIPTS_FILE 字段及自动解析写入逻辑

**【209服务器运维】**
- [x] 为使 Docker 容器端口可被外部访问，开放防火墙 19 个监听端口
- [x] 为解决 PgBouncer 认证失败，修正 userlist.txt 格式并添加 ignore_startup_parameters
```

## 日志模板

新建日志时使用以下结构：

```markdown
# 工作日志 - YYYY-MM-DD

## 日期信息

| 字段 | 内容 |
|------|------|
| 日期 | YYYY-MM-DD |
| 星期 | 星期X |
| 本周第几天 | 第 N 天 |

---

## 今日总结

### 完成事项 ✅

**[归属标识]**
- [x] 为{目的}，{具体操作与关键改动}

### 进行中事项 🔄

- [ ] 事项描述

### 待处理事项 ⏳

- [ ] 事项描述

---

## 任务进度更新

| 任务ID | 任务名称 | 今日进展 | 完成度变化 |
|--------|----------|----------|------------|
| T001 | 任务名 | 今天做了什么 | +X% |

---

## 问题与思考

### 遇到的问题

- 问题描述及解决方式

### 反思与改进

- 反思内容

---

## 明日计划

- [ ] 计划内容

---

## 备注

- 补充说明

---

## 附件/链接

- 相关文件路径或链接
```

## 周五特殊处理

如果今天是周五，将"明日计划"改为"下周计划"，内容应包含：
1. 本周未完成事项的延续
2. 下周重点任务安排
3. 需要跟进的问题

## 约束

- 所有日志内容必须基于实际对话，不能编造
- 日志文件必须位于 `logs/YYYY-MM/` 目录下，不能放在 `logs/` 根目录
- 追加时必须按归属分类整理，不能简单追加到末尾
- 已完成事项用 `- [x]`，未完成用 `- [ ]`
- 每条完成事项必须包含归属标识和目的描述
- 归属标识从文件路径自动推断，不确定时向用户确认
