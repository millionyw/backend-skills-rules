# 任务管理技能

## 描述

此技能用于帮助用户创建新任务、将复杂目标分解为可执行的子任务，并初始化任务元数据。

## 使用场景

### 正向触发 (When to use)

- 用户要求"创建新任务"、"立项"或"分解目标"。
- 用户提出创建新任务的需求
- 用户希望将一个目标分解为多个具体任务
- 用户需要初始化任务文件的元数据
- 需要在项目根目录的 `tasks/` 文件夹下初始化任务管理文档。

### 负向触发 (When NOT to use)

- 用户只是询问现有任务状态，不涉及文件创建或大规模修改。
- 用户明确要求按月份（如 2026-03.md）汇总任务时（除非该任务名称就是月份）。
- 用户请求查看任务清单（只读操作）。

## 指令

### 步骤一：提取核心要素
1. **识别项目名称**：从用户输入中提取最核心的动作或项目名，作为后续的文件名。
2. **确认目标与时限**：明确任务要达成的最终结果及截止日期。
   - **日期处理规则**：
     - **相对日期**（如"本周五"、"下周一"、"明天"、"后天"等）：必须根据当前日期计算出具体的 YYYY-MM-DD 格式日期
     - **月度模糊日期**（如"3月中下旬"、"月底"、"月初"等）：可以保留用户描述
     - **"前/后"后缀**：日期后可加"前"或"后"表示时间范围（如"2026-03-06前"）
   - **日期转换示例**：
     - "本周五" → "2026-03-06"（假设今天是 2026-03-03）
     - "本周五前" → "2026-03-06前"
     - "下周一" → "2026-03-09"
     - "明天" → "2026-03-04"
     - "3月中下旬" → "3月中下旬"（保留原样）
   - **重要提醒**：始终参考 <env> 中的当前日期进行计算
3. **确认任务类型**：明确用户要创建的任务类型（项目任务、日常任务、学习任务等）
4. **关键信息确认**：识别以下模糊信息并主动询问用户：
   - 涉及图片/附件的任务：询问具体内容或关键信息点
   - 跨月/长期任务：确认各阶段关键时间节点
   - 多人协作场景：确认各负责人及职责
   - 外部依赖：确认与外部系统或人员的依赖关系

### 步骤二：生成单一任务文件
1. **创建文件**：在 `tasks/` 目录下创建 `tasks/{任务/项目名称}.md` 文件。
2. **禁止拆分**：**严禁**为任何子任务创建独立的 `.md` 文件。所有信息必须保存在这一个文件中。
3. **使用任务模板格式**：在文件中按模板格式写入任务名称、状态、截止日期、优先级和完成度。
3. **注入元数据**：在文件顶部按模板格式写入任务 ID（T+年月日+序号）、优先级和状态。

### 步骤三：执行任务分解
1. **任务分层原则**：子任务必须按"用户/干系人视角的交付成果"划分，而非按内部实现步骤划分。
   - ✅ 正确示例："开发用户登录功能"（一个交付成果）
   - ❌ 错误示例：拆分为"设计数据库表"、"编写后端API"、"实现前端页面"（内部实现步骤）
2. 在同一文件中，将复杂目标拆解为逻辑清晰的子任务。
    a. 分析任务的组成部分
    b. 识别任务的依赖关系
    c. 确定执行顺序
3. 以 Markdown 列表或子表格的形式展现，确保父子任务的层级关系在视觉上可追溯。
4. **输出确认**：生成任务清单表格，包含：
   - 任务ID
   - 任务名称（必须是用户可感知的交付成果）
   - 状态（默认：待开始）
   - 截止日期
   - 优先级
   - 当前完成度

### 步骤四：输出反馈
1. 告知用户文件已创建，并列出该任务的唯一 ID。


## 约束

- **文件唯一性**：一个项目请求仅允许产出**一个** Markdown 文件。子任务必须以列表项形式嵌入，不得触发文件系统创建新文件
- **存储路径**：所有任务文件必须存放在项目根目录的 `tasks/` 文件夹下
- **命名规范**：文件名必须使用 `{任务/项目名称}.md`, 且任务名称不得重复
- **ID唯一性**：任务ID必须唯一，建议使用日期+序号格式，如 T2026030101
- **禁止执行任务**：创建任务后，不应主动执行任务中的开发、测试等实际操作。我的职责是记录和追踪任务，而不是完成它们。
